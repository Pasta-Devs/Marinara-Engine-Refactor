Architecture

Dependency direction is the main guardrail.

The app is layered so product behavior stays readable and privileged local work stays behind Tauri and Rust capability boundaries. The refactor is now organized around separate app shell, feature UI, shared frontend utilities, React-free engine modules, typed Tauri adapters, thin commands, and focused Rust capability crates.

UI Layer Detail

UI module Owns Important dependencies Boundary note
src/app/shell Application chrome, titlebar actions, sidebars, right panel, detail route registry, and the active surface slot. Feature surfaces, shared stores, generic UI components. Routes between product areas but should not own product rules.
features/modes The mode router that chooses conversation, roleplay, or game surface for the active chat. features/chats, features/roleplay, features/game. This is the intentional UI composition point for the three modes.
features/chats Conversation surface, chat input, message rendering, setup, settings, summaries, swipes, branches, galleries, and normal chat hooks. Chat engine services, generation hook, shared API adapters, shared stores. Owns conversation UI; currently also hosts reusable message pieces used by roleplay and game.
features/roleplay Roleplay surface, scene HUD, scene lifecycle controls, choices, encounters, and roleplay panels. Roleplay engine services, chat hooks/message components, visuals, storage and LLM adapters. Product behavior is roleplay-owned; UI still reuses chat message/input components.
features/game Game surface, setup, party UI, narration, maps, combat, inventory, journal, sessions, audio, game stores, and game API facade. Game engine services, shared API adapters, game assets, chat hooks/types for session messages. Game rules are separate; UI still reuses chat hooks and gallery/message types.
features/mari Professor Mari standalone assistant surface, local conversation state, connection/persona selectors, attachments, and Mari send flow. engine/mari, shared/api/mari-api, chat message component, connections/personas hooks. Not a chat, roleplay, or game mode; it is a shell-level assistant path.
features/game-assets, features/visuals, features/tracker, features/world-state Mode-adjacent UI for assets, sprites/weather/world display, trackers, and world state persistence. Shared API adapters, shared stores, game and roleplay surfaces. These are shared UI support areas; changes can affect more than one mode.
Library, provider, import, and integration features Characters, personas, lorebooks, presets, chat presets, connections, settings, imports, knowledge, gallery, Spotify, bot browser, onboarding. Storage/import/export/LLM/integration APIs, engine contracts, shared UI. They should expose feature hooks or facades instead of being reached through private component internals.

Ownership Rules

Layer Owns Must not own
React features Rendering, local UI state, user actions, feature hooks. Prompt rules, provider transport, storage formats, cross-mode orchestration.
TypeScript engine Product behavior, mode orchestration, prompt construction, generation flow. React imports, Tauri invokes, secrets, filesystem access, native network policy.
shared/api Typed Tauri adapter functions and channel/listen wrappers. Product decisions or mode-specific behavior.
Rust commands Input validation, DTO boundaries, command routing to capability services. Long product workflows, prompt semantics, UI decisions.
Rust crates Storage, path safety, provider transport, assets, imports, integrations. Mode state machines or UI feature logic.

Mode Separation Status

Area checked Status What is shared What to watch
engine/modes/chat Separated from roleplay and game at the product-rule layer. No direct sibling-mode imports were found. Contracts, engine shared helpers, generation/generation-core, repositories, and capability ports. Do not move roleplay scene lifecycle or game turn logic into chat services.
engine/modes/roleplay Separated from chat and game at the product-rule layer. No direct sibling-mode imports were found. Contracts, storage/LLM capability ports, generation, deterministic helpers, and visual asset helpers through UI layers. Scene orchestration stays here; shared transcript or parser helpers belong below modes.
engine/modes/game Separated from chat and roleplay at the product-rule layer. No direct sibling-mode imports were found. Contracts, generation, generation-core, repositories, shared parsers/scoring/text helpers, and capability ports. Game turns, game prompts, state repair, maps, mechanics, weather, time, and sessions stay game-owned.
features/chats Dedicated conversation UI, but not fully UI-isolated. Imports the roleplay scene banner for active scene controls and the game mode store for notification bubbles. These are UI bridge points. If they grow, move the shared contract into features/modes or src/shared.
features/roleplay Dedicated roleplay UI, but it reuses chat UI pieces. Uses chat hooks plus chat message, input, branch selector, overlays, echo chamber, summary popover, and chat-area types. This is component reuse, not roleplay using the chat engine. Reusable message primitives are candidates for a shared UI owner later.
features/game Dedicated game UI, but it reuses chat session hooks and message/gallery types. Uses chat hooks for session messages, chat-area types for shared display data, and the chat gallery drawer. Game rules remain in engine/modes/game. UI reuse should not become shared game/chat orchestration.
features/modes Intentionally coupled to all three mode surfaces. Imports chat, roleplay, and game surfaces because it is the UI router. Keep routing here; do not add prompt rules or mode state machines to the router.
Current answer: conversation, roleplay, and game are separated in the TypeScript engine where product behavior lives. They are not fully separated at the React UI component layer because roleplay and game reuse chat message/session UI, and chat has small roleplay/game bridge imports. Treat those as UI composition points, not permission to share mode orchestration.

Possible Docker Web Deployment

A VPS-hosted browser build is plausible, but it is not supported by the current tree as a first-class runtime. The main architecture conflict is not React or the TypeScript engine; it is that frontend capability access currently assumes Tauri invoke, Tauri channels, Tauri file URL conversion, and desktop plugins. A Docker web mode should be added as a second host adapter, not as browser-only conditionals scattered through feature code.

What Works With The Current Re-org

Area Assessment Why
React feature surfaces Mostly reusable. Most UI already depends on feature hooks, shared UI, and src/shared/api rather than importing Rust or Tauri directly.
TypeScript engine Reusable. The engine is React-free and receives capability objects for storage, LLM, integrations, and events. This is the right shape for both desktop and web hosts.
Rust capability crates Reusable. Storage, assets, LLM transport, security, and integrations are already below Tauri command facades, so an HTTP host can call the same lower services.
Tauri command facades Not directly reusable as web routes. They are shaped as #[tauri::command] handlers. The shared service code below them should be reused, while HTTP routes should be thin sibling facades.
Frontend API transport Needs re-org. src/shared/api/tauri-client.ts is currently the single command transport. Browser mode needs a parallel fetch/SSE/WebSocket transport selected by runtime target.
Desktop plugins and local files Needs explicit browser fallbacks. File dialogs, window titlebar controls, opener behavior, asset protocol URLs, and local file paths have different browser semantics.

Required Modular Seams

Seam Target shape Do not do
Runtime target Add one host selector such as VITE_TARGET=tauri and VITE_TARGET=web, exposed through a small runtime-env helper. Do not test window.__TAURI__ inside every feature component.
Command transport Split command execution behind a transport interface: Tauri uses invoke; web uses same-origin HTTP. Do not duplicate storage-api.ts, llm-api.ts, and other adapters into separate web copies.
Streaming Keep llm-api.ts provider-neutral and adapt Tauri Channel to web SSE or WebSocket events. Do not make generation code know whether tokens came from Tauri channels or browser streams.
Local assets Move asset URL resolution behind an asset URL adapter. Tauri can keep convertFileSrc; web should return authenticated or scoped HTTP URLs. Do not leak server filesystem paths to the browser.
File picking and uploads Use browser <input type="file"> and multipart/upload endpoints in web mode; keep Tauri dialog paths for desktop imports where needed. Do not make browser mode accept arbitrary client filesystem paths.
Desktop-only UI Wrap titlebar controls, folder openers, and OS window behavior in small host adapters with no-op or browser equivalents. Do not let desktop window commands throw during normal browser rendering.
Rust host Add a web/server binary that serves dist/ and exposes HTTP routes calling the same Rust capability services as Tauri commands. Do not put product behavior into the HTTP routes; keep them thin like Tauri commands.
Build and CI Build both targets: Tauri desktop and Docker web. Add a web typecheck/build path so browser-only regressions are caught. Do not rely on desktop CI to prove browser mode still works.

Browser Mode Conflict Areas

Tauri invoke

Most persistence, imports, exports, LLM calls, integrations, Professor Mari, and media commands currently enter through invokeTauri. Browser mode needs a transport swap at that layer.

Streaming channels

LLM streaming currently uses Tauri channels. Web mode needs SSE or WebSocket events with the same frontend event contract.

Local asset URLs

Desktop can convert local paths with Tauri. Web mode must serve managed assets through HTTP URLs and should never expose raw server paths.

File dialogs

Tauri dialog paths do not exist in a remote browser. Imports and uploads need browser file inputs and upload endpoints.

Window and opener plugins

Titlebar controls, folder openers, and native window behavior need browser-safe no-ops or equivalents.

Local integrations

Haptics, Spotify OAuth callbacks, TTS, and bot/browser utilities need per-integration decisions because the server may be remote while the device is local.

Recommendation

The feature can fit the current architecture if the next step is a host-adapter re-org, not a feature-by-feature patch. The first useful cut is a web runtime that supports storage, LLM generation, chat/roleplay/game, managed asset reads, imports through browser uploads, and Professor Mari. Desktop-only behavior can be marked unsupported or degraded until each integration gets a web contract.

That means no broad product re-org is needed in src/engine. The needed modularity is mostly at src/shared/api, src/shared/api/local-file-api.ts, desktop plugin call sites, and Rust command facades. If those seams are added cleanly, a Docker deployment can serve the Vite build and Rust API from one process without changing the chat, roleplay, or game engine owners.

Honest cost: every future PR touching Tauri APIs, asset URLs, uploads, streaming, or native integrations needs a browser-mode review. CI should build both desktop and web targets, or web regressions will slip through.

When To Add A Module

  1. Put UI rendering under the owning feature.
  2. Put product rules under the owning engine mode or engine service.
  3. Put reusable pure helpers below the modes.
  4. Put Tauri wrappers in src/shared/api.
  5. Put privileged local work in Rust commands and crates.

If a file needs to import sideways across modes or private features, move the shared concept down before adding the import.
